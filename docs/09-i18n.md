# 09 — Internationalization (i18n)

Added in v2 (BACKLOG stories S19–S20). The app is fully localized in **5 languages**: English (`en`, default and fallback), French (`fr`), Spanish (`es`), Italian (`it`), German (`de`). The language follows the **browser language** — there is no language picker UI and no persistence (explicit v2 non-feature).

Two halves, two stories:

- **Backend (S19)**: localized *content* — body names, descriptions, compositions, fun facts — served by `GET /api/bodies?lang=`.
- **Frontend (S20)**: localized *chrome* — every label, button, hint and unit in the React UI, via a pure `domain/i18n/` module.

## Locale detection (frontend, `domain/i18n/locale.ts`)

```ts
export type Locale = "en" | "fr" | "es" | "it" | "de";
export const SUPPORTED_LOCALES: readonly Locale[] = ["en", "fr", "es", "it", "de"];
export const DEFAULT_LOCALE: Locale = "en";

/** Pure: "fr-FR" → "fr", "es-419" → "es", "pt-BR" → "en" (fallback). */
export function resolveLocale(navigatorLanguage: string | undefined): Locale;
```

Rule: take the **primary subtag** of `navigator.language` (everything before the first `-`), lowercased; if it is in `SUPPORTED_LOCALES`, use it; otherwise `DEFAULT_LOCALE`. Resolved **once at boot** in the react layer and passed down (the domain function itself never reads `navigator` — it takes the string as input, so it stays pure and testable).

The resolved locale is sent to the API as `?lang=<locale>` (doc 02). The frontend always sends a valid code, so the API's `400` on unknown codes is never hit in normal operation.

## API contract (backend)

See doc 02 for the full endpoint spec. Summary: `GET /api/bodies?lang=<code>` with `code ∈ {en, fr, es, it, de}`; omitted → `en`; any other value → `400 { "error": "Invalid lang" }`. The DTO shape (doc 02) is **unchanged** — only the *values* of `name`, `info.description`, `info.composition`, `info.funFact` are localized.

Backend data layout:

- `en` is served from the canonical data files (`data/bodies.ts` names + `data/bodyInfo.ts`, i.e. doc 03 Table 4 verbatim).
- Each other language is one file `apps/backend/src/data/localized/<lang>.ts` exporting:

```ts
export const localizedBodies: Record<string, { name: string; info: BodyInfo }> = { /* all 29 ids */ };
```

A completeness test (doc 02, test 21) asserts every language covers all 29 ids with non-empty `name`, `description`, `composition`, `funFact`.

## Translation policy for the long texts

**The English texts of doc 03 Table 4 are canonical.** The translated `description` / `composition` / `funFact` strings (29 bodies × 3 fields × 4 languages) are **faithful translations of Table 4**, written and reviewed during story S19, and live **only** in the `data/localized/<lang>.ts` files — they are deliberately *not* duplicated in the docs. Rules:

- Same facts, same numbers, same tone; no added or dropped information.
- Numbers keep the doc 03 digit-formatting (thin-space grouping, e.g. `2 100 km/h`).
- Proper nouns (Olympus Mons, Verona Rupes, Herschel, E ring…) are kept unless the language has a well-established exonym.
- If a Table 4 text changes later, all four translations must be updated in the same commit.

## UI strings (frontend, `domain/i18n/strings.ts`)

export function t(locale: Locale, key: UiKey, params?: Record<string, string>): string;

## Layering rules (doc 04 applies)

- `domain/i18n/` is **pure**: no `navigator`, no React, no fetch. Fully unit-tested (`resolveLocale`, `t`, dictionary completeness).
- The react layer resolves the locale once at boot (`resolveLocale(navigator.language)`), passes it to `api/fetchBodies(lang)` and to every component that renders text.
- The three layer renders no text — it needs no i18n.
